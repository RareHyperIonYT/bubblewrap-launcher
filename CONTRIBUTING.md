# Contributing to Bubblewrap Launcher

Thank you for your interest in contributing! Before you open anything, please read this document.

## Policies

### No AI-generated content.

This project does **not** accept AI-generated pull requests, issues, code, and documentation.

If you use AI tools for brainstorming or reference, that is your choice, but the final contribution must be your own work and judgment. Do not submit content that was generated, heavily rewritten, or meaningfully completed by an AI system.

If a contribution appears AI-generated, it may be closed or rejected without review.

## Reporting bugs

Before opening an issue:

- Check that you're on the latest commit.
- Search existing issues to avoid duplicates.

A good bug report includes:

- What you did
- What you expected to happen
- What actually happened
- Your OS, relevant versions (Rust, Node, Firejail, and Wine if applicable), and any error output.

This project is in early development and things will break often. Low-quality reports will be closed without response.

## Suggesting features

Check the [ROADMAP.md](./ROADMAP.md) first as your idea may already be planned.

Feature requests are welcome as issues, but keep in mind:

- This project has a clear direction; suggestions that don't fit that direction will probably end up not implemented.

- Early-stage projects move fast. A feature request might become irrelevant before it's implemented.

Be specific. "Improve the UI" is not a feature request.

## Contributing code

> Note: This project is in early, unstable development (v0.1.0). Architecture may change frequently. Coordinate before putting significant effort into a PR.

#### Before you start

Open an issue or comment on an existing one to discuss the change. This avoids wasted effort if the approach doesn't fit the project.

#### Development setup

See the [README.md](./README.md) for prerequisites and setup instructions.

#### Submitting a PR

- Keep changes focused. One thing per pull-request.
- Write your own commit messages that actually describe what changed.
- Test your changes before submitting.
- If your PR fixed a bug, reference the issue.
- If it's a non-trivial change, explain your reasoning. Not just *what* but *why*.

PRs that touch areas still under active design may be held or redirected based on upcoming architectural decisions.
